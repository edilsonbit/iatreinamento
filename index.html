Uma mudança prática de operação: o dev para de digitar tudo do zero e passa a pilotar contexto, revisar código e validar com testes.

Como ensinar seu padrão para a IA

O que é esse arquivo?

  • Um arquivo .github/copilot-instructions.md na raiz do repositório.
  • O Copilot lê automaticamente antes de gerar qualquer código ou commit.
  • Define padrões de commit, branch, nomenclatura e testes do seu time.

Como funciona na prática

  • Dev pede ao Copilot para gerar uma mensagem de commit.
  • Copilot consulta o arquivo e aplica o padrão do time automaticamente.
  • Resultado: feat(auth): 20260415_1234 add token rotation

Como criar

Crie o arquivo .github/copilot-instructions.md na raiz do repositório e defina os padrões. O Copilot aplica automaticamente em todos os devs do time que usarem o Copilot naquele repo.

Exemplo de regras que você pode definir

  • Formato de commit e tipos aceitos (feat, fix, refactor...)
  • Padrão de branch naming
  • Template de PR e checklist obrigatório
  • Regras de nomenclatura de métodos e variáveis
  • Avisos de segurança (nunca incluir tokens, validar entradas)

Glossário rápido para nivelar linguagem

Termos de fluxo

  • diff: diferença entre versão antiga e nova do código.
  • commit: pacote pequeno de alterações versionadas.
  • branch: linha isolada de trabalho.
  • PR: pedido de revisão para integrar mudanças.
  • review: validação técnica do PR.

Termos de entrega

  • issue: item rastreável no backlog.
  • hotfix: correção urgente em produção.
  • rollback: retorno para versão estável anterior.
  • rebase: reaplicar commits sobre base atualizada.
  • squash: juntar vários commits em um.

Padrão de commit rastreável

Estrutura de prompt para devs

Template (modelo abstrato)

Você é um engenheiro sênior.
Contexto: [stack/módulo].
Objetivo: [entrega].
Regras: [arquitetura, segurança, testes].
Saída: [código, testes, riscos, checklist].

Exemplo prático preenchido

Você é um engenheiro sênior Node.js.
Contexto: API REST com Express, módulo de autenticação, banco PostgreSQL.
Objetivo: criar endpoint POST /login que valida email e senha, retorna JWT e lança erro 401 em credencial inválida.
Regras: sem SQL raw, usar bcrypt para comparação de senha, retornar apenas mensagem genérica de erro, incluir teste unitário com cenário de sucesso e falha.
Saída: código do controller + teste + lista de riscos de segurança.

Funciona melhor quando:

  • Há contexto real do repositório.
  • A tarefa é quebrada em passos pequenos.
  • Você pede alternativas com trade-off.

Falha quando:

  • Prompt vago e sem restrição técnica.
  • Não existe critério de aceite definido.
  • Sugestão é aceita sem validação humana.
